# Warvote 프로젝트 브랜치 관리 가이드

브랜치는 프로젝트의 메인 코드 흐름(`main`)에서 뻗어 나온 **'평행 우주'**와 같습니다. 메인 코드를 건드리지 않고 자유롭게 실험하고 기능을 개발한 뒤, 검증이 끝나면 다시 하나로 합치는 방식입니다.

## 1. 왜 브랜치를 써야 하나요?

1.  **안전성:** 메인 코드(`main`)는 항상 '실행 가능한 상태'여야 합니다. 브랜치를 쓰면 미완성 코드가 메인에 섞이는 것을 막을 수 있습니다.
2.  **독립성:** 팀원 A는 'CRM 개발', 팀원 B는 '지도 시각화'를 서로 방해받지 않고 동시에 진행할 수 있습니다.
3.  **코드 리뷰:** 기능을 합치기 전에 팀원들이 코드를 미리 검토하고 승인하는 과정을 거칠 수 있습니다.

## 2. 권장되는 브랜치 전략 (GitHub Flow)

Warvote 프로젝트에서는 가장 단순하고 효율적인 **GitHub Flow** 방식을 권장합니다.

1.  **main (메인):** 항상 출시 가능한 최신 코드가 있는 곳입니다. 직접 수정하지 않습니다.
2.  **feature/기능명 (피처 브랜치):** 새로운 기능을 만들 때마다 생성하는 가지입니다.
    *   예: `feature/sms-tracking`, `feature/user-auth`
3.  **hotfix/버그명:** 급하게 수정해야 하는 오류가 있을 때 생성합니다.

## 3. 브랜치 작업 프로세스

### 3.1 새로운 기능 개발 시작
먼저 새로운 브랜치를 만들고 이동합니다.
```bash
git checkout -b feature/new-logic
```

### 3.2 작업 및 커밋
브랜치 안에서 코드를 수정하고 저장합니다.
```bash
git add .
git commit -m "feat: 유권자 데이터 필터링 로직 추가"
```

### 3.3 GitHub에 브랜치 올리기
내 컴퓨터에만 있는 브랜치를 온라인 저장소에 올립니다.
```bash
git push origin feature/new-logic
```

### 3.4 Pull Request (PR) 생성 및 합치기 (Merge)
GitHub 웹사이트에서 `Pull Request`를 생성하여 "내가 만든 이 가지를 main에 합쳐도 될까?"라고 팀원들에게 물어봅니다. 리뷰가 끝나면 `Merge` 버튼을 눌러 합칩니다.

### 3.5 브랜치 삭제
합치기가 끝난 브랜치는 깔끔하게 삭제합니다.
```bash
git branch -d feature/new-logic  # 로컬 브랜치 삭제
git push origin --delete feature/new-logic # 원격 브랜치 삭제
```

## 4. Antigravity에게 시키는 법

명령어가 복잡하다면 저(Antigravity)에게 다음과 같이 요청하세요.

*   **브랜치 생성:** `"새로운 CRM 기능을 위해 feature/crm-v1 브랜치 만들고 거기로 이동해 줘."`
*   **브랜치 확인:** `"지금 내가 어떤 브랜치에서 작업하고 있어?"`
*   **머지(Merge) 요청:** `"지금 feature 브랜치 작업을 다 끝냈는데, main 브랜치로 합쳐줄 수 있어?"`
*   **상태 정리:** `"작업 끝난 브랜치들 다 삭제하고 main으로 돌아가 줘."`

---

## 5. 주의사항
*   **Merge 전 Pull 필수:** 브랜치를 합치기 전이나 작업을 시작할 때는 항상 `git pull origin main`을 통해 최신 상태를 유지하세요.
*   **브랜치 이름:** 누구나 알 수 있게 명확하게 지어주세요. (`update1` (X) -> `feature/login-ui-fix` (O))
